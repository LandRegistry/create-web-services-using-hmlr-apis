Testing the Submit an application to change the register APIs

Overview

Sandbox is a test system that allows you to test your full end-to-end scenarios in a secure, controlled environment. It offers a wide range of test scenarios and provides realistic results, closely matching the live environment.

Sandbox supports safe, repeatable experimentation to help reduce risk and ensure changes are well-tested before deployment. This enables you to try new configurations, understand potential impacts, and catch issues early without impacting your live environment.

For its initial release, we are focusing on delivering the sandbox for the ‘Submit an application to change the Land Register APIs’.

As we develop the sandbox, it will include our other APIs. In the meantime, the existing test stubs will run alongside the sandbox for a period but will eventually be phased out. 

About the sandbox

Unlike test stubs, which provide a small amount of test scenarios, sandbox allows you to create unlimited test scenarios to optimise integration testing.  Sandbox simulates real test scenarios by providing test title data.

In sandbox, there is no limit to the type of application you can submit. You can submit any kind of application with any combination of transactions and get realistic results.

Sandbox simulates everything up to the submission or rejection of an application. There are no further stages after this point. In addition, for successful submissions, it also allows you to do the following:

  • testing of requisitions
  • under certain conditions, it will produce notifications for a requisition

Validation is usually quicker in sandbox than in the live system. This is to allow integrators to run automated tests in a reasonable time.

In sandbox, you may find that validation results are available in a few seconds. But, in the live system, you may find that it takes a few minutes. However, you should not expect validation to complete immediately, and you must poll the notification API for validation results.

How to access the sandbox

Prerequisites

Existing customers

Before you can use the sandbox, you will need your account credentials. To get your credentials, you will need a TLS client certificate and a password.

If don't already have a certificate or password, email Channel Partners to request them.

Once you have your credentials, follow these steps:

  1. Install or configure certificates (installation will vary based on your test setup).
  2. Receive username and password for sandbox.
  3. Confirm the environment is reachable by making an API request.

Internal developers

If you are an internal developer, you should already have a test account set up. You won't need a TLS certification, and you can bypass the authenticate step.

You will receive a document upload URL once you have logged into the internal test account.

Start using our APIs

View our APIs from the ‘Submit an application to change the Land Register APIs’ to learn more about how they work.

To view the guidance, select the API you want to test in sandbox from the list.

How does the API work

Overview of the end-to-end process

First, gather the necessary information from your customer and ensure all information has been checked. This should include the documents, in one of the accepted formats (such as PDF, JPG, GIF or TIFF).

Once this is done, follow these steps:

  1. For each document, you must: 
    • assign an ID for the document - this is a string and should be unique within the application, for example: “document-1” or “document-947927345”
    • calculate the sha256 hash of the binary data of the file (in the sandbox this must be one of the supplied sample files)
    • obtain a document URL from the /v1/documents/url endpoint 
    • store the document URL in your system against the document record 
  2. Generate the application model
  3. Submit the application 

If you receive errors

You should handle any errors that occur, usually by presenting it to the end user of your system for resolution. If you receive a status indicating that validation is in progress, you should poll for notifications or status updates every five minutes using the notifications API.

If validation fails, you should handle the errors and warnings appropriately.

Status updates after submission

When the document application proceeds to ACCEPTED_AWAITING_PRIORITY, it means the application has been accepted outside of working hours and will receive priority at the beginning of the next working day.

In the sandbox, to allow for convenient testing, title numbers ending in 12 are always out of hours. Other titles are always in hours, regardless of the time of day or day of the week. Out of hours applications will become priority protected within a few minutes.

If the application proceeds to ACCEPTED_PRIORITY_PROTECTED, your application has been accepted and has been assigned a priority. 

How to submit an application in the sandbox

At a glance

  1. Request and retrieve the document upload URL
  2. Upload the document and receive the document reference
  3. Add the document reference/ID into the application
  4. Submit the application for validation
  5. Confirm submission is done
  6. Find out if validation succeeded or failed

Example of submitting an application in the sandbox

Imagine you wish to submit a simple charge against a single title.

To do this, you would follow these steps:

  1. Collect the relevant test title data. This should include all the details needed to submit the application, such as:
    • the title number
    • the existing proprietors
    • the applicants
    • the details of the charge to be added
    • it should also include binary data for any documents required, in one of the accepted formats such as PDF, JPG, GIF or TIFF
  2. Upload the documents (in this example, a Charge deed or CH1 form).

    To do this you need to obtain the document upload URL by calling by the API of your choice. To obtain the document upload URL you must supply:
    • the size
    • the sha256 hash of the document file
    • other information specified in the documentation
    You must then record the identifier returned by the API of your choice. Once done, upload the document (in the sandbox, this should be one of the supplied sample files).
  3. Record the document ID in your system
  4. Submit a new application by calling the API of your choice. You need to send a JSON object, and the documents you mention should refer to the document IDs you got earlier.
  5. Check if the application has passed or failed validation. If it has failed, review that specific API from the Submit an application to change the Land Register APIs. Validation in sandbox follows the same as validation rules as those used in the live system

Test data

Test data within a sandbox environment is designed to mimic real-world data, allowing you to run various scenarios and validate their systems without affecting live data. This ensures a realistic and safe testing experience that limits risk.

Test data is meant to represent real data so that it can help you test live outcomes as closely as possible.

When submitting an application in a live environment, you must supply title numbers, addresses, and names of proprietors or lenders that match data in HM Land Registry systems.

For the sandbox, we supply the following test data– using real titles will not work. You will have your own test data environment. This means you are able to use this test title data without it affecting anything.

Titles

Download the list of generated titles. This list includes where they have charges, the chargee (for example a lender, or charge proprietor).

All sandbox titles start with 'TT'. The numbers are made up to supply synthetic data and meet specific conditions, and don’t match real titles.

There are groups of synthetic titles with similar restrictions and numbers of proprietors, but they differ in names and addresses.

For example, TT000100 to TT000119 are freehold titles with one proprietor, where the proprietor's address matches the property's address.

Multiple similar titles are provided to allow integrators to run parallel tests without causing priority issues to the day list.

Sandbox-generated titles have various properties, including charges, with or without restrictions, and additional lender obligations. 

Similarly, sample titles may be shown with or without charges, restrictions, or notices. But these are generated without using the register.

Existing charges, where these are MD-Ref charges (with a pre-approved mortgage deed reference number), the charge will have the form of a synthetic MD-Ref charge. And the charge (the mortgage company) will refer to a synthetic lender name corresponding to that MD Ref.

Addresses

Addresses used in test data are real addresses and are taken from the Ordnance Survey (OS) sample data set, which is available for download on the OS website. This is to provide a more realistic test experience.

These addresses are not linked to real people, restrictions, or charges.

Where a company or organisation name appears in the OS address data, this will be removed and not appear in the synthetic title data.

For example, integrators may use existing address validation which references Royal Mail PAF or OS data. This allows them to test their entire system, including their internal validation against PAF/OS data, along with the BG Sandbox validation.

The title data does not reference any title held by the land registry. For example, a title with a specific address may be shown as a closed title, development title, leasehold, or freehold in the sandbox. This has no relation to real titles held by HM Land Registry.

Any relation between synthetic title data in the sandbox and real data is coincidental as we do not use the register data to generate synthetic data.

Lender data

Charges (mortgages and loan agreements) are usually identified by an MD Reference (MD Ref), which identifies the lender.

The sandbox provides synthetic MD References which can be used when submitting applications in the sandbox. None of the MD References in the sandbox are real. Similarly, the lenders recorded on the title data are not real.

MD references

When submitting applications, use this list of MD Reference charges. These all begin with the characters MD9T.

At the time of publishing, none of the test data MD reference values relate to real live MD references, as none start with ‘MD9T’. However, this may change. You must not reject MD reference values in your live system if they match the ‘MD9T’ pattern.

Names

Names of natural persons

Where a person’s name is required in test data, these are taken from a list of fake names.  These names are not taken from any living person, or organisation, or trademark, or copyrighted work.

The surnames are taken from European cities and towns, and the forenames are taken from common names of gemstones, trees, flowers, and culinary herbs. For example, TT000410 has proprietors “Cedar Cagnes-sur-Mer” and “Fern Manchester”.

When submitting an application against a test title, you should consult the supplied title data to find which name to use.

Any relation to any living person, organisation, work, trademark, or any intellectual property is coincidental.

Creating and using test data

The test data supplied in the sandbox was created following these test data principles. Any test data you submit to the sandbox should follow these principles.

  1. Trademark and copyright compliance

    Test data should not be based on any trademark, copyrighted work, or the name of any current company or corporate body (except for statutory bodies which may be required to be present) or any living person.

  2. Realistic-looking data

    Test data should look as much as possible like real data. For example, names should be a mix of single and double-barrelled names, with one, two, three, and four forenames. But you should never use real data or commercially sensitive data.

  3. Easily identifiable as test data

    Test data should be reasonably easy to recognise as test data. This is to help ensure that no test data is accidentally used in a live system.

  4. Name patterns

    If you see a name where the forenames are gemstones, flowers, herbs, or trees, and the surname is the name of a town or city in Europe, it is possible that this is test data.

  5. Organisation and corporate body names

    If you see a name of a corporate body (such as a limited company or building society) which contains the word “test,” it is possible (and likely) that this is test data. For example, “Ruby Yew Test Mortgages PLC” is a mocked lender name, which may be used when submitting test applications with MD Reference “MD9TAB” – which is also mock data.

  6. MD reference numbers

    If you see an MD Reference number beginning with “MD9T” (for example, MD9TAB), then it is very likely that it is test data. This prefix was chosen because very few live MD Reference numbers begin with MD9, and none (at the time of writing) begin with MD9T (where T is chosen to represent the word “test”).

    You must not validate MD Reference numbers based on not matching that pattern in the live system.

    In the sandbox, all MD Reference numbers match the pattern “MD9T.” Therefore, in the sandbox, it is fine to insist on matching that pattern – it is only that we cannot guarantee that pattern will never exist in the live system.

Reporting test data errors

If you see any data that is not referenced in this guide, it is likely we made an error.

Report any errors in deployment to Channel Partners, or to the Service Desk if errors are affecting the live environment.

Testing scenarios

There are an unlimited amount of possible test scenarios. It is not possible to provide all test scenarios for this reason.

We have provided an initial list of test scenarios for the first release of sandbox. More test scenarios will be developed and added at a later stage.

Before you start the test scenarios, familiarise yourself with the test title data. The title you choose should be suitable for the test you wish to run. For example, for a DJP (death of joint proprietor), the title must have two proprietors.

Data policy

Data in sandbox is stored in the database for 24 hours.

You can run an overnight test without waiting until the next morning.

Validation rules

The sandbox follows the same validation rules as the ones used in the live system, and it supplies almost-complete coverage of the validation phases.

To learn about the API validation rules, review the API guidance in the Submit an application to change the Land Register APIs.
