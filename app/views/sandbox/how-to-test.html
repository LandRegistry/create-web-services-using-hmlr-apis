Testing the Submit an application to change the register APIs

For this initial release of the sandbox, we are focusing on delivering the sandbox for the Submit an application to change the Land Register APIs.

As we develop the sandbox, it will include our other APIs. In the meantime, the existing test stubs will run alongside the sandbox for a period but will eventually be phased out.

Help us improve this page and the developer pack by providing feedback via this form.

Overview

Sandbox is a test system that allows you to test your scenarios in a secure, controlled environment. It offers a wide range of test scenarios and provides realistic results, closely matching the live environment.

Sandbox supports safe, repeatable experimentation to help reduce risk and ensure changes are well-tested before deployment. This enables you to try new configurations, understand potential impacts, and catch issues early without impacting your live environment.

About the sandbox

Unlike test stubs, which provide a small amount of test scenarios, sandbox allows you to create unlimited test scenarios to optimise integration testing.  Sandbox simulates real test scenarios by providing test title data.

In sandbox, you can submit any kind of application with any combination of transactions and get realistic results, provided it is found within the test title data.

Sandbox simulates everything up to the submission or rejection of an application. There are no further stages after this point.

Validation has been made quicker in sandbox than in the live system, as it utilises artificial time constraints. For example, out of hours requests are returned instantly. This is to allow integrators to run tests in a reasonable time.

In sandbox, you may find that validation results are available in a few seconds. But, in the live system, you may find that it takes longer. However, you should not expect validation to complete immediately, and you must poll the notification API for validation results.

How to access the sandbox

If you are an existing user of our APIs, you must:

  • reuse your current test TLS certificate
  • request access credentials (a username and password) from the Channel Partners (channelpartners@landregistry.gov.uk). You may have already recieved this as part of onboarding to use the sandbox environment

This sandbox is a closely managed test environment for the initial release, so access is controlled.

Validation rules

The sandbox follows the same validation rules as the live system, and it supplies coverage of all validation phases, with the help of test data.

To learn about the API validation rules, review the API guidance in the Submit an application to change the Land Register APIs.

Test data

Test data within a sandbox environment is designed to mimic real-world data, allowing you to run various scenarios and validate their systems without affecting live data. This ensures a realistic and safe testing experience that limits risk.

Test data is meant to represent real data so that it can help you test live outcomes as closely as possible.

When submitting an application in a live environment, you must supply title numbers, addresses, and names of proprietors or lenders that match data in HM Land Registry systems.

For the sandbox, we supply the following test data– using real titles will not work. You will have your own test data environment. This means you are able to use this test title data without it affecting anything.

Limitations of test data

Currently, the test data only supports freehold titles.

Titles

Download the list of generated titles. This list includes where they have charges, the chargee (for example a lender, or charge proprietor).

All sandbox titles start with 'TT'. The numbers are made up to supply synthetic data and meet specific conditions, and don’t match real titles.

There are groups of synthetic titles with similar restrictions and numbers of proprietors, but they differ in names and addresses.

For example, TT000100 to TT000119 are freehold titles with one proprietor, where the proprietor's address matches the property's address.

Multiple similar titles are provided to allow integrators to run parallel tests without causing priority issues to the day list.

Sandbox-generated titles have various properties, including charges, with or without restrictions, and additional lender obligations. 

Similarly, sample titles may be shown with or without charges, restrictions, or notices. But these are generated without using the register.

Existing charges, where these are MD-Ref charges (with a pre-approved mortgage deed reference number), the charge will have the form of a synthetic MD-Ref charge. And the charge (the mortgage company) will refer to a synthetic lender name corresponding to that MD Ref.

Addresses

Addresses used in test data are real addresses and are taken from the Ordnance Survey (OS) sample data set, which is available for download on the OS website. This is to provide a more realistic test experience.

These addresses are not linked to real people, restrictions, or charges.

Where a company or organisation name appears in the OS address data, this will be removed and not appear in the synthetic title data.

For example, integrators may use existing address validation which references Royal Mail PAF or OS data. This allows them to test their entire system, including their internal validation against PAF/OS data, along with the BG Sandbox validation.

The title data does not reference any title held by the land registry. For example, a title with a specific address may be shown as a closed title, development title, leasehold, or freehold in the sandbox. This has no relation to real titles held by HM Land Registry.

Any relation between synthetic title data in the sandbox and real data is coincidental as we do not use the register data to generate synthetic data.

Lender data

Charges (mortgages and loan agreements) are usually identified by an MD Reference (MD Ref), which identifies the lender.

The sandbox provides synthetic MD References which can be used when submitting applications in the sandbox. None of the MD References in the sandbox are real. Similarly, the lenders recorded on the title data are not real.

MD references

When submitting applications, use this list of MD Reference charges. These all begin with the characters MD9T.

At the time of publishing, none of the test data MD reference values relate to real live MD references, as none start with ‘MD9T’. However, this may change. You must not reject MD reference values in your live system if they match the ‘MD9T’ pattern.

Names

Where a person’s name is required in test data, these are taken from a list of fake names.  These names are not taken from any living person, or organisation, or trademark, or copyrighted work.

The surnames are taken from European cities and towns, and the forenames are taken from common names of gemstones, trees, flowers, and culinary herbs. For example, TT000410 has proprietors “Cedar Cagnes-sur-Mer” and “Fern Manchester”.

When submitting an application against a test title, you should consult the supplied title data to find which name to use.

Any relation to any living person, organisation, work, trademark, or any intellectual property is coincidental.

Reporting test data errors

If you see any data that is not referenced in this guide, it is likely we made an error.

Report any errors in the sandbox to Channel Partners (channelpartners@landregistry.gov.uk).

 

Testing scenarios

We have supplied an example of a successful and unsuccessful application submission.

Before you start the test scenarios, familiarise yourself with the test title data. The title you choose should be suitable for the test you wish to run. For example, for a DJP (death of joint proprietor), the title must have two proprietors.

Example of submitting an application in the sandbox

Imagine you wish to submit a simple charge against a single title.

To do this, you would follow these steps:

  1. Collect the relevant test title data. This should include all the details needed to submit the application, such as:
    • the title number
    • the existing proprietors
    • the applicants
    • the details of the charge to be added
    • it should also include binary data for any documents required, in one of the accepted formats such as PDF, JPG, GIF or TIFF
  2. Upload the documents (in this example, a Charge deed or CH1 form).

    To do this you need to obtain the document upload URL by calling by the Send a document API. To obtain the document upload URL you must supply:
    • the size
    • the sha256 hash of the document file
    • other information specified in the documentation
    You must then record the identifier returned by the Send a document API. Once done, upload the document (this should be one of the supplied sample files).
  3. Record the document ID in your system.
  4. Submit a new application to change the Land Register by calling the Submit an application API. You need to send a JSON object, and the documents you mention should refer to the document IDs you got earlier.
  5. Check if the application has passed or failed validation. If it has failed, review that specific API from the Submit an application to change the Land Register APIs. Validation in sandbox follows the same as validation rules as those used in the live system
