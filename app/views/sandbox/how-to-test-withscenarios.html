Testing the Submit an application to change the Land Register APIs

For this initial Beta release of the sandbox, we are focusing on simulating APIs from the Submit an application to change the Land Register APIs up until the point of application acceptance. The sandbox does not currently support any further stages.

Help us improve this page and the developer pack by providing feedback via this form. Your feedback is valuable to us and will directly inform future improvements to the sandbox.

Overview

The sandbox is a secure, controlled test system that provides your developers with an integration environment. It enables repeatable experimentation so you can safely explore configuration, understand potential impacts, and identify issues early. The sandbox does not affect HM Land Registry’s live production register and will not incur fees.

The sandbox simulates real test scenarios by providing dummy data, called  test title data. It simulates everything up to acceptance or rejection of an application. The sandbox does not currently support further stages - applications won't be processed, completed, or cancelled. Correspondence and completion documents will not be issued for downloading.

You may find that processing times vary and application validation will not always match the live system.

How to access the sandbox

If you are an existing user of our APIs, you must:

  • reuse your current test TLS certificate or, if required, request a new SSL test certificate from Channel Partners (channelpartners@landregistry.gov.uk)
  • request access credentials (a username and password) — also from Channel Partners. You may have already received this as part of onboarding to use the sandbox environment
  • call the APIs from the address https://bgtest.landregistry.gov.uk/bg2sandbox/api

This sandbox is a closely managed test environment for the initial release, so access is controlled.

Start using our APIs

View our APIs from the Submit an application to change the Land Register APIs to learn more about how they work. To view the guidance, select the API you want to test in the sandbox from the list.

In the sandbox, use version 1.0 of the APIs.

The APIs currently supported in the sandbox are:

Send a document API (Version 1.0)

Allows the request of an upload URL, uploading documents and provides partial security checks.

The sandbox currently supports a restricted file upload size of 100KB.

Submit an application API (Version 1.0)

Allows for submission of an application up to the point of acceptance or rejection. It will not include cancelled and completed.

Notifications API (Version 1.0)

Provides all notification types that are sent up to the point of an application being accepted or rejected.

Application information API (Version 1.0)

Provides detailed information about a specific application, such as getting the latest application status including validation failures. Will not include any correspondence.

Limitations

As this is an early release of the sandbox, some functionality is not yet available. We have provided a list of known limitations, but this is not an exhaustive list.

We’ll continue to improve and resolve these limitations as we develop the sandbox. Please let us know if you encounter any unexpected limitations.

Currently, the sandbox is unable to:

  • upload files larger than 100KB
  • view post-accepted statuses, which includes completed and cancelled
  • reply to a requisition with documents and messages
  • access requisition documents, completion documents and other correspondence
  • receive notifications that occur after the acceptance or rejection of an application

Validation rules

The aim of the sandbox is to follow the same validation rules as the live system. It will also cover all the validation phases. However, at present, there may be some differences between validation in the sandbox and the live service.

Test data

Test data in the sandbox is designed to mimic real‑world data, so you can run tests without affecting live systems. This provides a realistic testing experience while reducing risk.

When submitting an application in a live environment, you must supply title numbers, addresses, and names of proprietors or lenders that match data in HM Land Registry systems.

For the sandbox, we supply the following test data – using real titles will not work. You will have your own test data environment.

! Warning Only use the provided test data in the sandbox. Do not upload real application data.

Limitations of test data

Currently, the test data only supports freehold titles.

Reporting test data errors

If you see any data that is not referenced in this guide, it is likely made in error.

Report any errors in the sandbox to Channel Partners (channelpartners@landregistry.gov.uk).

 

Titles

Download the list of generated titles. This list includes some title numbers where there are existing charges (such as lenders, and charge proprietors).

All sandbox titles start with 'TT'. The numbers are made up to supply synthetic data and meet specific conditions, and don’t match real titles.

There are groups of synthetic titles with similar restrictions and numbers of proprietors, but they differ in names and addresses.

For example, TT000100 to TT000119 are freehold titles with one proprietor, where the proprietor's address matches the property's address.

Sandbox-generated titles have various properties, including charges, with or without restrictions, and additional lender obligations. 

Similarly, sample titles may be shown with or without charges, restrictions, or notices. But these are generated without using the register.

Existing charges, where these are MD-Ref charges (with a pre-approved mortgage deed reference number), the charge will have the form of a synthetic MD-Ref charge. And the charge (the mortgage company) will refer to a synthetic lender name corresponding to that MD Ref. See lender data for more about existing charges where there are MD References.

Addresses

Addresses used in test data are real addresses and are taken from the Ordnance Survey (OS) sample data set, which is available for download on the OS website. This is to provide a more realistic test experience.

These addresses are not linked to real people, restrictions, or charges.

Where a company or organisation name appears in the OS address data, this will be removed and not appear in the synthetic title data.

For example, integrators may use existing address validation which references Royal Mail PAF or OS data. This allows them to test their entire system, including their internal validation against PAF/OS data, along with the BG Sandbox validation.

The title data does not reference any title held by the land registry. For example, a title with a specific address may be shown as a closed title, development title, leasehold, or freehold in the sandbox. This has no relation to real titles held by HM Land Registry.

Any relation between synthetic title data in the sandbox and real data is coincidental as we do not use the register data to generate synthetic data.

Lender data

Charges (mortgages and loan agreements) are usually identified by an MD Reference (MD Ref), which identifies the lender.

The sandbox provides synthetic MD References which can be used when submitting applications in the sandbox. None of the MD References in the sandbox are real. Similarly, the lenders recorded on the title data are not real.

When submitting applications, use this list of MD Reference charges. These all begin with the characters MD9T.

At the time of publishing, none of the test data MD reference values relate to real live MD references, as none start with ‘MD9T’. However, this may change. You must not reject MD reference values in your live system if they match the ‘MD9T’ pattern.

Names

Where a person’s name is required in test data, these are taken from a list of fake names.  These names are not taken from any living person, or organisation, or trademark, or copyrighted work.

The surnames are taken from European cities and towns, and the forenames are taken from common names of gemstones, trees, flowers, and culinary herbs. For example, TT000410 has proprietors “Cedar Cagnes-sur-Mer” and “Fern Manchester”.

When submitting an application against a test title, you should consult the supplied title data to find which name to use.

Any relation to any living person, organisation, work, trademark, or any intellectual property is coincidental.

Documents

The sandbox supports file uploads of up to 100KB. You can upload more than one document, as long as each file does not exceed this limit.

Testing scenarios

Before you start using the sandbox, familiarise yourself with the test title data. The title you choose should be suitable for the test you wish to run. For example, for a DJP (death of joint proprietor), the title must have two proprietors.

! Warning You should only use the provided test title data and not any real data.

Example of submitting an application in the sandbox

Imagine you wish to submit a simple charge against a single title.

To do this, you would follow these steps:

  1. Collect the relevant test title data. This should include all the details needed to submit the application, such as:
    • the title number
    • the existing proprietors
    • the applicants
    • the details of the charge to be added
    • the binary data for any documents required, in one of the accepted formats such as PDF, JPG, GIF or TIFF
  2. Upload the documents (in this example, a Charge deed or CH1 form).

    To do this you need to obtain the document upload URL by calling the Send a document API. To obtain the document upload URL you must supply:
    • the size
    • the sha256 hash of the document file
    You must then record the identifier returned by the Send a document API. Once done, upload the document.
  3. Record the document ID in your system.
  4. Submit a new application to change the Land Register by calling the Submit an application API. You need to send a JSON submission payload, and the documents you mention should refer to the document IDs you got earlier.
  5. Take note of the application ID returned after you lodge your application.
  6. Use the application request ID at the Application information API to check if the application has passed or failed validation. If the application has failed, review the OpenAPI specification for Submit an application API or Send a Document API for any validation issues
